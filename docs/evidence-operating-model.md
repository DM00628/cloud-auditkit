# Evidence Operating Model

## Purpose
This document explains how evidence should be requested, reviewed, referenced, and retained for compliance work performed through this repository.

## Principles
- Store evidence references in version control
- Avoid placing sensitive raw evidence in public repositories
- Link to approved evidence locations when raw artifacts are restricted
- Require reviewer validation for high impact controls

## Minimum evidence fields
- Control ID
- Artifact name
- System or boundary
- Source owner
- Collection date
- Reviewer
- Review outcome
- Storage location

## Review workflow
1. Request evidence from the control owner
2. Record the request in a GitHub issue or project item
3. Add the evidence reference to the control mapping or POA&M
4. Review for completeness, currency, and signatures if required
5. Track gaps as findings or POA&M items

## Retention
Follow your organizational retention requirements for audit evidence, assessment notes, and remediation records.
